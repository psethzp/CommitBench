## The biggest problems

### 1. Raw success is 100% everywhere

This is the biggest reviewer-killer. If every model/system has 100% raw success, reviewers will say the benchmark is not measuring real task-solving difficulty. It currently looks like the agent always emits one of the terminal actions and the scaffold marks that as success.

So the paper must not say:

> “We evaluate realistic task success.”

It can say:

> “We evaluate least-effect certification among successful traces in a controlled source-backed adapter suite.”

But for EACL main, I strongly recommend adding a **native-fidelity subset** where terminal success is actually computed from native final-state/tool-state predicates. τ-bench is explicitly about tool-agent-user interaction with domain APIs, policies, final database-state evaluation, and pass^k; ToolSandbox emphasizes stateful tool execution and intermediate/final milestones. Your current adapter uses those as source-backed substrates, but it does not yet look like native execution. ([arXiv][2])

### 2. EffectGuard does not really beat PROJ_GUARD

Current result:

| Comparison  | Strict-excess rate |
| ----------- | -----------------: |
| PROJ_GUARD  |            7.4358% |
| EFFECTGUARD |            7.3940% |
| Difference  |          0.0419 pp |

That is basically a tie. Your own guard-tie report says only **3/7,168** paired guarded units differ in verifier verdict. So do **not** claim EffectGuard is meaningfully better than the projection guard from current results.

Root cause: current `PROJ_GUARD` already performs most lower-effect substitutions. It is not a clean “projection-only” baseline. You need either:

1. rerun with a corrected `PROJ_GUARD_V2` that only blocks projected violations and does **not** globally substitute lower-effect traces; or
2. demote EffectGuard to “reference controller” and make the paper about the kernel/verifier, not online policy superiority.

For EACL main, I would do option 1.

### 3. Current verifier is not yet a complete kernel verifier

The current verifier compares successful traces generated by the systems in the same task/model/seed/regime group. That is useful, but it is not the same as proving nondominance against the **full admissible action graph**.

Reviewers may ask:

> “How do you know the witness/frontier is complete and not just an artifact of the three policies you happened to run?”

You need a cheap offline fix: **enumerate admissible action sequences** for the compact action grammar, compute the full nondominated frontier, and relabel observed traces against that frontier. No GPU needed.

This is probably the highest ROI fix in the whole plan.

### 4. Projection baselines are not SOTA baselines

Your current projection table is useful, but call them:

```text

CORE_DFA projection
MINISCOPE_PERMISSION projection
CONTRACT_MENU_CMTF projection

MODERNSTACK_PROJECTION

```

Do **not** call them faithful SOTA systems. Current `CORE_DFA` accepts all traces and is numerically identical to `FINAL_STATE`, so it is not a serious full-path baseline yet.

### 5. Lattice sensitivity looks fake/degenerate

Every lattice variant gives exactly the same strict-excess rate:

```text

```

That looks suspicious. Either fix it so tradeoffs can change incomparability/dominance, or drop it from the main paper.

### 6. CEGAR needs targeted stress

Current CEGAR is strong for `outbox`, `memory_cache`, and `user_visible_exposure`, but not for:

```text

contract_artifact_hash
virtual_clock
compensation_or_payment_hold
```

Those have zero label-changing collisions. That does not mean they are irrelevant; it means the current scaffold did not exercise them. Add targeted stress cases or do not make strong claims about those fields.
